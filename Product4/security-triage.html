The Problem

Applying for a Principal PM role in cybersecurity services without direct security domain experience creates a specific credibility gap. Talking about governance architecture and human-in-the-loop design is one thing. Demonstrating it in a security context is another. The goal was to close that gap with a working artifact, not a slide deck.

The Approach

The starting point was the same framework I apply to every AI product decision: define the decision first, then the workflow, then the governance model, then the technology.

In a security triage context, the core decision pattern is consistent: should this threat be acted on, escalated, deferred, or monitored, and who owns that call? That framing was familiar. I spent 20 years building decision-support systems around asset inventory, dependency mapping, risk classification, and prioritization dashboards for CIOs. The data layer in security operations - CMDB records, asset exposure, vulnerability scoring, risk-ranked queues - is the same architecture applied to a different domain. The governance and workflow design transfer directly. What I Built

The dashboard integrates three threat data sources through a single governed workflow.

🔴 CVE Triage
NIST NVD + CISA KEV
Live vulnerability feed with exploitability cross-reference

Live CVE feed from the NIST National Vulnerability Database, filtered to CVSS 9.0+ critical vulnerabilities published in the last 90 days, cross-referenced against the CISA Known Exploited Vulnerabilities catalog. The KEV integration was a deliberate design decision — CVSS score measures theoretical severity, but KEV status tells an analyst whether threat actors are actively exploiting the vulnerability today. Surfacing that distinction in the UI changes the triage question from "how bad is this" to "how urgent is this right now." Analyst actions: Approve & Route, Escalate, or Override & Defer with mandatory written rationale.

📧 Phishing
Email Analysis
AI-assisted phishing email triage

Phishing email analysis using a labeled research corpus demonstrating AI risk scoring and triage workflow governance. AI analyzes email content for threat indicators, risk level, and recommended action. Analyst confirms block, escalates to SOC, or releases as false positive with written rationale. In production this connects to a live email security gateway feed. Analyst actions: Block & Quarantine, Escalate to SOC, or Release — False Positive with mandatory override rationale.

🌐 URL Indicators
Threat Intelligence
Active threat indicator classification and response

URL threat indicator feed with severity classification across phishing, malware distribution, ransomware, credential harvesting, and account takeover categories. AI generates threat analysis including affected systems and mitigation steps. Analyst approves block or initiates investigation before any action executes. Analyst actions: Block Immediately, Investigate, or False Positive with mandatory rationale.

📋 Audit Log
All Decisions
Full accountability and override analysis

Every analyst decision logged across all three threat sources — approval, escalation, and override — with timestamp, AI recommendation, human decision, and written rationale. Override analysis surfaces where AI and analyst judgment diverge, enabling model quality improvement over time without retraining. Full CSV export for compliance reporting.

Key Design Decisions

Each decision was made for an operational reason, not a technical one.

KEV Integration — Added Post-Build

The initial build surfaced CVEs by CVSS score alone. That is not how experienced analysts prioritize. A CVSS 10.0 with no known exploit in the wild is lower priority than a CVSS 7.5 that ransomware groups are actively using. Surfacing exploitability context in the expander header - before the analyst opens the record - reduces cognitive load and improves decision quality. That is a PM decision, not an engineering decision.

Mandatory Override Rationale

Override rationale was made mandatory rather than optional for the same reason it is mandatory in the Portfolio Rationalization Advisor: optional rationale produces no audit trail and no feedback signal. Mandatory rationale creates both. When an analyst disagrees with the AI recommendation and there is no written rationale, the disagreement is invisible and unlearnable.

Three-Tab Structure

The three-tab structure mirrors how security teams organize their work — vulnerability management, email security, and threat intelligence are distinct operational functions with distinct decision owners. A single unified queue would obscure that structure rather than support it. Workflow design follows organizational structure, not engineering convenience.

Date-Filtered CVE Feed

The NVD API without a date filter returns historically prominent CVEs, not operationally current ones. Filtering to CVSS 9.0+ published in the last 90 days, then cross-referencing KEV, produces a queue that reflects what a real SOC analyst would actually be working on today - not a historical archive.

Technical Architecture

Stack

  • Python + Streamlit - rapid full-stack deployment without a separate frontend
  • Anthropic Claude API (claude-sonnet) - AI triage summaries, threat analysis, and phishing risk scoring
  • NIST NVD API - live CVE feed filtered to CVSS 9.0+ critical, last 90 days
  • CISA KEV API - daily-refreshed known exploited vulnerabilities catalog
  • GitHub + Streamlit Community Cloud - production deployment, CI/CD via git push

Governance Controls

  • Human review required before any action executes across all three threat surfaces
  • Mandatory written rationale for every override or false positive decision
  • Full audit log with timestamp, AI recommendation, human decision, and rationale
  • Override analysis - surfaces where AI and analyst judgment diverge
  • CSV export for compliance reporting

Data Sources

  • NIST NVD - authoritative CVE database, live API, date-filtered and severity-filtered
  • CISA KEV - confirmed active exploitation catalog, daily refresh
  • Phishing research corpus - labeled dataset demonstrating triage workflow and governance architecture; production connects to live email security gateway
  • URL threat indicator feed - severity-classified active malicious URL indicators

Deliberate Choices

  • On-demand AI generation: Triage summaries generated on analyst request rather than pre-scored at ingestion - demonstrates the interaction pattern while controlling API costs at demo scale. Production would pre-generate on ingestion.
  • Date + severity + KEV filtering: Three-layer CVE filtering produces an operationally realistic queue rather than a historical artifact list.
  • Same governance pattern as Portfolio Advisor: Deliberate - demonstrates pattern reusability across domains, not one-off capability.

What It Demonstrates

The governance pattern is the same as the Portfolio Rationalization Advisor - retrieve context, generate recommendations, require human review, execute, and log outcomes. That pattern transfers across domains because the underlying problem is the same: AI surfaces relevant context faster than a human can, but humans own the decision and the accountability.

Built and deployed in under 48 hours, this case study demonstrates how enterprise AI governance and decision-support experience can transfer into a cybersecurity services environment. The domain required active learning. The underlying PM pattern did not: define the decision, structure the workflow, manage risk, keep humans accountable, and create the audit trail required for trusted AI adoption.
